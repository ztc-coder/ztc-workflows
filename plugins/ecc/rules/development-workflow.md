# 开发工作流

> 扩展 [common/git-workflow.md](./git-workflow.md)，定义 git 操作前开发流程。

先分级，再研究、规划、TDD、审查、提交前检查。

## 1. 任务分级策略

按序判定：

1. 命中高风险 → 大型或高风险
2. 未命中高风险且满足小型快路径 → 小型低风险
3. 其余 → 中型

仅在新风险或影响扩大时升级。不因 agent、skill 或工具扩大范围。

| 等级 | 判定条件 | 默认流程 |
|------|----------|----------|
| 小型低风险 | 文档、typo、局部样式、简单配置、无行为变化；或 ≤1-2 文件、仅单模块内部、不涉及 API/网络/文件系统/数据库/权限边界，可局部验证 | 最小定位 → 最小修改 → 最小验证 |
| 中型 | 未命中高风险，且不满足小型低风险条件 | 简短计划 → 实现 → 针对性验证 → 按需审查 |
| 大型或高风险 | 命中高风险；或跨模块/多服务、架构调整、数据或安全语义变化、部署/生产配置、难回滚 | planner → 分阶段实现 → 验证计划 → 代码审查+专项检查 |

高风险条件：

- 架构调整、大规模重构、多模块或多服务变更
- 数据库迁移、数据破坏风险、数据写入语义变化
- 认证、授权、权限判断、会话处理
- 支付、加密、密钥、敏感数据
- 文件上传、文件系统写入、路径处理
- CI/CD、部署、生产配置
- 外部请求目标变化、命令执行参数变化
- 性能敏感、安全敏感或难回滚

普通输入校验、局部 API 契约微调、纯类型层变更默认不升高风险，但须补验证。

**默认不运行生产构建命令，如 `npm run build`、`pnpm build`、`yarn build`；除非用户明确要求或项目规则允许。**
**默认不运行`npm run lint`、`npm run dev`等命令**

## 功能实现工作流程(小型低风险任务默认跳过该流程只做最小修复 -> 最小验证)

0. **研究与复用** *(任何新实现前必须执行)*
   * **优先进行 GitHub 代码搜索：** 在编写任何新代码之前，先运行 `gh search repos` 和 `gh search code` 以查找现有的实现、模板和模式。
   * **其次查阅库文档：** 在实现之前，使用 Context7 或主要供应商文档来确认 API 行为、包的使用以及版本特定的细节。
   * **仅在以上两者不足时使用 Exa：** 在 GitHub 搜索和主要文档之后，再使用 Exa 进行更广泛的网络研究或探索。
   * **检查包注册中心：** 在编写工具代码之前，先搜索 npm、PyPI、crates.io 和其他注册中心。优先选择经过实战检验的库，而不是自己动手实现。
   * **寻找可适配的实现：** 寻找能解决 80% 以上问题的开源项目，以便进行分叉、移植或封装。
   * 如果经过验证的方法能满足需求，优先采用或移植该方法，而不是编写全新的代码。

1. **先规划**
   * 使用 **planner** 智能体来创建实施计划
   * 编码前生成规划文档：PRD、架构、系统设计、技术文档、任务列表
   * 识别依赖项和风险
   * 分解为多个阶段

2. **TDD 方法**
   * 使用 **tdd-guide** 智能体
   * 先编写测试（RED）
   * 实现代码以通过测试（GREEN）
   * 重构（IMPROVE）
   * 验证 80% 以上的覆盖率

3. **代码审查**
   * 编写代码后立即使用 **code-reviewer** 智能体
   * 解决 CRITICAL 和 HIGH 级别的问题
   * 尽可能修复 MEDIUM 级别的问题

4. **提交与推送**
   * 详细的提交信息
   * 遵循约定式提交格式
   * 提交信息格式和 PR 流程请参阅 [git-workflow.md](git-workflow.md)
