# 重构清理

在每个步骤进行测试验证的情况下，安全地识别并移除死代码。

## 第一步：检测死代码

根据项目类型运行分析工具：

| 工具 | 检测内容 | 命令 |
|------|--------------|---------|
| knip | 未使用的导出、文件、依赖 | `npx knip` |
| depcheck | 未使用的 npm 依赖 | `npx depcheck` |
| ts-prune | 未使用的 TypeScript 导出 | `npx ts-prune` |
| vulture | 未使用的 Python 代码 | `vulture src/` |
| deadcode | 未使用的 Go 代码 | `deadcode ./...` |
| cargo-udeps | 未使用的 Rust 依赖 | `cargo +nightly udeps` |

若无可用工具，使用 Grep 查找零 import 的导出：
```
# 查找导出，然后检查它们是否在任何地方被 import
```

## 第二步：分类发现

将发现按安全层级排序：

| 层级 | 示例 | 操作 |
|------|----------|--------|
| **SAFE** | 未使用的工具函数、测试辅助函数、内部函数 | 放心删除 |
| **CAUTION** | 组件、API 路由、中间件 | 验证无动态 imports 或外部消费者 |
| **DANGER** | 配置文件、入口点、类型定义 | 操作前先调查 |

## 第三步：安全删除循环

对每个 SAFE 项：

1. **运行完整测试套件** — 建立基线（全部绿色）
2. **删除死代码** — 使用 Edit 工具精确移除
3. **重新运行测试套件** — 验证无破坏
4. **若测试失败** — 立即用 `git checkout -- <file>` 回滚并跳过此项
5. **若测试通过** — 继续下一项

## 第四步：处理 CAUTION 项

删除 CAUTION 项前：
- 搜索动态 imports：`import()`、`require()`、`__import__`
- 搜索字符串引用：配置中的路由名、组件名
- 检查是否从公共包 API 导出
- 验证无外部消费者（若已发布则检查依赖方）

## 第五步：整合重复

移除死代码后，查找：
- 近似重复函数（>80% 相似）— 合并为一个
- 冗余类型定义 — 整合
- 无附加价值的包装函数 — 内联
- 无意义的重导出 — 移除间接层

## 第六步：总结

报告结果：

```
死代码清理
──────────────────────────────
已删除：  12 个未使用函数
          3 个未使用文件
          5 个未使用依赖
已跳过：  2 项（测试失败）
节省：    ~450 行已移除
──────────────────────────────
所有测试通过 PASS:
```

## 规则

- **绝不在运行测试前删除**
- **每次一个删除** — 原子性变更使回滚容易
- **不确定时跳过** — 保留死代码好过破坏生产环境
- **清理时不重构** — 分离关注点（先清理，后重构）
